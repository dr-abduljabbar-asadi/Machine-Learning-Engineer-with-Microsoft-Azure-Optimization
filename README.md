## Summary
**In 1-2 sentences, explain the problem statement: e.g "This dataset contains data about... we seek to predict..."**

This dataset contains records from a phone-based bank marketing campaign, including client demographics, contact details, and campaign attributes. We seek to predict whether a client will subscribe to a bank term deposit (`y = yes/no`) — a binary classification problem.

**In 1-2 sentences, explain the solution: e.g. "The best performing model was a ..."**

The best performing model was an Azure AutoML **VotingEnsemble** with an accuracy of about **0.917**, which narrowly outperformed the HyperDrive-tuned Scikit-learn Logistic Regression (accuracy about **0.915**).

## Scikit-learn Pipeline
**Explain the pipeline architecture, including data, hyperparameter tuning, and classification algorithm.**

The pipeline is defined in `train.py`. It loads the Bank Marketing data from the source URL into a `TabularDataset` with `TabularDatasetFactory.from_delimited_files()`, cleans and one-hot encodes the categorical columns, and converts the target into a 0/1 label. A **Logistic Regression** classifier (Scikit-learn) is then trained with two tunable hyperparameters — `C` (inverse regularization strength) and `max_iter` (maximum iterations). **Azure HyperDrive** wraps the script in a `ScriptRunConfig` and searches the hyperparameter space to maximize **accuracy**, sampling `--C` from a uniform range (0.08 – 0.1) and `--max_iter` from a discrete set (25, 50, 100, 200).

**What are the benefits of the parameter sampler you chose?**

I chose **random sampling** (`RandomParameterSampling`). It supports both discrete and continuous hyperparameters, explores the search space efficiently without exhaustively testing every combination (unlike grid search), and works well with early-stopping policies — so compute isn't wasted on unpromising runs.

**What are the benefits of the early stopping policy you chose?**

I used a **Bandit policy** (`slack_factor=0.1`, `evaluation_interval=2`, `delay_evaluation=5`). It leaves the first 5 runs alone so nothing is cut off prematurely, then checks every 2 intervals and terminates any run whose accuracy falls outside the top 10% of the best run so far. This saves significant time and compute by stopping weak runs early without hurting the quality of the final model.

## AutoML
**In 1-2 sentences, describe the model and hyperparameters generated by AutoML.**

AutoML ran a classification task with 5-fold cross-validation and selected a **VotingEnsemble** (accuracy ~0.917), which blends several models (e.g. XGBoost, LightGBM, SGD, RandomForest) using weighted soft voting. Across the ensembled algorithms, the regularization strengths and maximum iterations were the most influential hyperparameters, and AutoML also applied automatic featurization and scaling (e.g. `MaxAbsScaler`).

## Pipeline comparison
**Compare the two models and their performance. What are the differences in accuracy? In architecture? If there was a difference, why do you think there was one?**

The two models were very close in accuracy: the HyperDrive Logistic Regression reached about **0.915** while the AutoML VotingEnsemble reached about **0.917** — a small edge to AutoML. The architectures differ significantly: HyperDrive tuned a single, manually chosen model (Logistic Regression) over a predefined search space, whereas AutoML automatically selected both the algorithm and its hyperparameters and ultimately produced an ensemble of models. The likely reason for the difference is that the ensemble pools predictions from several diverse models via weighted voting, which tends to be more robust and slightly more accurate than any single tuned linear model.

## Future work
**What are some areas of improvement for future experiments? Why might these improvements help the model?**

- **Address class imbalance** — AutoML's data guardrails flagged the target as imbalanced (the minority class is much smaller). Applying resampling or class weights would help the models detect the minority class and improve metrics beyond raw accuracy (e.g. AUC, F1).
- **Widen the hyperparameter search** — Test a broader range of `C` and `max_iter` and increase the number of HyperDrive runs to explore more of the space, which may uncover better settings.
- **Try different primary metrics** — Optimizing for AUC_weighted instead of accuracy would give a fairer picture on imbalanced data.
- **More feature engineering** — Building new features and dropping unhelpful ones may give the models a stronger signal to learn from.
